# Manifest spec

The AutoArchDX manifest is a JSON file generated by the scanner and consumed by the renderer, the GitHub Action, and any other tooling that builds on AutoArchDX. It is the single source of truth for the architecture graph at a given point in time.

The manifest lives at `.autoarch/manifest.json` in the repository root. It is a generated file — do not edit it by hand. Changes should be made by modifying annotations or YAML definitions and re-running the scanner.

The formal JSON Schema is at [`../schema/manifest.schema.json`](../schema/manifest.schema.json). This document is the narrative explanation of that schema.

---

## Envelope

The top level of the manifest is the envelope. It identifies the manifest format, the project, and the scanner that produced it.

```jsonc
{
  "version": "1.0",
  "scanner_version": "0.1.0",
  "generated": "2026-03-19T05:00:00Z",
  "project": "wfai-platform",
  "root": "./",
  "repo_url": "https://github.com/org/repo",
  "description": "WFAI scoring platform",

  "nodes": [...],
  "edges": [...],
  "groups": [...],
  "warnings": [...]
}
```

**Envelope fields:**

| Field | Required | Description |
|-------|----------|-------------|
| `version` | yes | Manifest schema version. |
| `scanner_version` | yes | Semver of the scanner that produced this manifest. |
| `generated` | yes | ISO 8601 timestamp of when the scan ran. |
| `project` | yes | Project name. Defaults to the repository directory name if not configured. |
| `root` | yes | Relative path to the repository root. Almost always `"./"`. |
| `repo_url` | no | Repository URL. Used by the renderer to generate source file links. |
| `description` | no | Short human description of the project. |

For forked repositories, the envelope also includes an `upstream` block. See [fork model](./fork-model.md).

## Nodes

`nodes` is an array of component objects. Each node represents one logical architectural component, which may have been declared by one annotation, multiple annotations, a YAML definition, or some combination.

```jsonc
{
  "id": "scoring-engine",
  "type": "platform",
  "status": "built",
  "label": "Scoring engine",
  "description": "Weighted fields, explanation factors",
  "tags": ["core", "go"],
  "group": "orchestration-layer",
  "layout": { "x": 80, "y": 390 },
  "source": "annotation",
  "provenance": { ... }
}
```

**Node fields:**

| Field | Required | Description |
|-------|----------|-------------|
| `id` | yes | Stable slug. Explicitly set or generated as a content hash. |
| `type` | yes | `platform` `api` `db` `client` `datasrc` `partner` `group` |
| `status` | yes | `built` `partial` `planned` `undef` |
| `label` | yes | Display name. |
| `description` | no | Short subtitle. |
| `tags` | no | Merged union of tags from all contributing sources. |
| `group` | no | Group membership. String for a single group, array of strings for multiple. |
| `layout` | no | `{x, y}` renderer position hint. |
| `source` | yes | `annotation` `yaml` `inferred` `both`. |
| `provenance` | yes | See [provenance model](./provenance-model.md). |

**Provenance shape by source:**

When `source` is `annotation` and the component was declared by a **single annotation**, `provenance` is a single annotation provenance object.

When `source` is `annotation` and the component was declared by **multiple annotations** (same `id`, different files or symbols), `provenance` is an array of annotation provenance objects — one per contributing annotation. This is the expected shape for any non-trivial component.

When `source` is `yaml`, `provenance` is a single YAML provenance object.

When `source` is `both`, `provenance` is an object with two keys: `annotation` (object or array) and `yaml` (object).

See [provenance model](./provenance-model.md) for full field reference.

## Edges

`edges` is an array of connection objects. Each edge represents a connection between two nodes — declared via annotation or YAML, or inferred from the import graph.

```jsonc
{
  "id": "e:scoring-engine->bigquery-warehouse",
  "from": "scoring-engine",
  "to": "bigquery-warehouse",
  "kind": "declared",
  "bidir": false,
  "label": "ingress",
  "provenance": { ... }
}
```

**Edge fields:**

| Field | Required | Description |
|-------|----------|-------------|
| `id` | yes | Generated. Format: `e:<from>-><to>`. Multiple edges between the same pair append a numeric suffix: `e:a->b:2`. |
| `from` | yes | Source node ID. |
| `to` | yes | Target node ID. |
| `kind` | yes | `declared` `inferred` `override` |
| `bidir` | yes | Boolean. |
| `label` | no | Edge label shown in the renderer. |
| `description` | no | Longer explanation. Not shown in renderer. |
| `provenance` | yes | See [provenance model](./provenance-model.md). |

**Edge kinds:**

`declared` — explicitly stated in `@arch:connects` or a YAML `connections` entry. Represents architectural intent. Renders as a solid line.

`inferred` — observed by the scanner in the import or call graph. Represents structural reality. Carries `confidence` in provenance. Renders as a dashed line, opacity scaled by confidence.

`override` — explicitly suppressed via `@arch:override deny:`. The suppression is auditable — override edges appear in the manifest but are not rendered by default. Available in the renderer's audit mode.

---

## Groups

`groups` is an array of logical container objects. Groups render as dashed bounding boxes. They have no edges and are not nodes in the graph.

```jsonc
{
  "id": "orchestration-layer",
  "label": "Orchestration layer",
  "description": "Core processing components",
  "members": ["run-lifecycle", "schema-config", "run-cache", "confidence-signal"],
  "layout": { "x": 40, "y": 310, "width": 700, "height": 400 },
  "provenance": { ... }
}
```

**Group fields:**

| Field | Required | Description |
|-------|----------|-------------|
| `id` | yes | Stable identifier. |
| `label` | yes | Display name. |
| `description` | no | Optional subtitle. |
| `members` | yes | Array of node IDs. Merged from all sources — `@arch:component group:...` and YAML `groups[].members` both contribute. |
| `layout` | no | `{x, y, width, height}`. |
| `provenance` | yes | See [provenance model](./provenance-model.md). |

## Warnings

`warnings` is an array of diagnostic entries. Warnings are never fatal — the scanner always produces a complete manifest.

```jsonc
{
  "code": "SOT_CONFLICT",
  "severity": "warn",
  "message": "Field 'status' conflict on component 'bls-oews': yaml=built, annotation=planned",
  "node_id": "bls-oews",
  "edge_id": null,
  "fields": ["status"],
  "resolution": "annotation",
  "provenance": {
    "source": "annotation",
    "file": "internal/data/bls.go",
    "line": 8,
    "symbol": "blsOEWSClient",
    "symbol_kind": "variable",
    "commit": "abc123def456",
    "branch": "main",
    "author": "john.bear"
  }
}
```

**Warning fields:**

| Field | Required | Description |
|-------|----------|-------------|
| `code` | yes | Machine-readable warning code. |
| `severity` | yes | `warn` — human action may be needed. `info` — informational confirmation. |
| `message` | yes | Human-readable description. |
| `node_id` | no | Related node, if applicable. |
| `edge_id` | no | Related edge, if applicable. |
| `fields` | no | Field names involved (e.g. for `SOT_CONFLICT`). |
| `resolution` | no | Which source won when a conflict was resolved. |
| `provenance` | no | For annotation-related warnings, always points to the annotation. |

---

## Warning reference

### Source-of-truth

| Code | Severity | Cause | Action |
|------|----------|-------|--------|
| `SOT_CONFLICT` | `warn` | Conflicting fields, no `!` on either side. Annotation values used. | Review annotation at given file/line. Resolve with `!id:` or `@arch:override allow:*`. |
| `SOT_OVERRIDE` | `info` | One side declared `!`. Resolved cleanly. | No action needed. |
| `SOT_DUAL_OVERRIDE` | `warn` | Both sides declared `!`. Annotation values used. | One side should remove `!`. Coordinate between architect and engineer. |
| `SOT_DEFERRED` | `info` | Annotation declared `allow:*`. YAML values used. | No action needed. |

### References

| Code | Severity | Cause | Action |
|------|----------|-------|--------|
| `BROKEN_REF` | `warn` | Connection references an ID not found in any source. Edge not created. | Check `to:` or `from:` for typos. Ensure target is declared. |
| `AMBIGUOUS_REF` | `warn` | Label matches more than one component. Edge not created. | Replace label reference with explicit `id`. |
| `DUPLICATE_ID` | `warn` | Two YAML entries declare the same `id`. First definition used. | Remove or rename the duplicate. |

### Implementation

| Code | Severity | Cause | Action |
|------|----------|-------|--------|
| `MISSING_ID` | `warn` | `@arch:component` has no explicit `id` and a matching YAML definition exists. | Add `id:<slug>` matching the YAML definition's `id`. |
| `MISSING_IMPL` | `warn` | YAML defines a component with status other than `undef` and no annotation exists. | Annotate the implementation when built, or set `status: undef` in YAML. |
| `ORPHANED_NODE` | `info` | A component has no edges of any kind. | Consider whether it should be connected, or is intentionally standalone. |

### Drift

| Code | Severity | Cause | Action |
|------|----------|-------|--------|
| `STALE_ANNOTATION` | `info` | A contributing symbol's file has been modified more than `stale_annotation_threshold` commits since `introduced_at`. | Review the annotation for accuracy. Update or assert `!id:` to confirm it is still correct. |

### Environment

| Code | Severity | Cause | Action |
|------|----------|-------|--------|
| `SHALLOW_CLONE` | `info` | Repository is a shallow clone. `introduced_at` fields may be incomplete. | Use `fetch-depth: 0` in GitHub Action checkout. |

## Manifest stability

The manifest is designed to be diffable and idempotent.

**Stable IDs.** Node IDs are explicit slugs wherever possible. Generated IDs (content hashes) are stable as long as the annotation doesn't move. A manifest diff caused by a real architectural change looks distinct from one caused by a file rename.

**Sorted output.** The scanner sorts all arrays deterministically: `nodes` and `groups` by `id`, `edges` by `from+to`, `warnings` by `code+node_id`. This prevents spurious diffs from reordering.

**Idempotent.** Running the scanner twice on the same codebase produces identical manifests (byte-for-byte, modulo the `generated` timestamp). The `generated` field is the only thing that changes between identical scans.

These properties make the GitHub Action's diff-and-PR workflow reliable. A manifest diff means something architectural actually changed.

## Configuration

The scanner reads `.autoarch/config.yml` if present:

```yaml
# .autoarch/config.yml
project: wfai-platform
repo_url: https://github.com/org/repo
description: "WFAI scoring platform"

scanner:
  root: ./
  exclude:
    - vendor/
    - node_modules/
    - "**/*_test.go"
  min_inferred_confidence: 0.2
  stale_annotation_threshold: 20
  languages:
    - go
    - typescript
```

All configuration fields are optional. The scanner operates with sensible defaults when no config file is present.
